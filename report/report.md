# Getting the sniffer to work


Contrary to previous years, according to the professor, getting the sniffer to work was relatively pain free by following Nordic's documentation.

For 802.15.4 packets, Nordic provides firmware that captures the 802.15.4 packets and outputs it to Serial ports, which can then be interfaced with a Python class for the sniffer that Nordic also provides. The firmware is installed using the nRF Connect desktop app by Nordic, which contains an app for installing firmware files onto the dongle. With this class, we can call the extcap_capture() method, which will record the packets and store them in a pcap file. Nordic also claims that this firmware is compatible with Wireshark, given that you install a plugin they provide, however it was not possible to get this mechanism to work.

For BLE packets, the Wireshark plugin did work. Nordic provides a CLI utility for it's chips and devices, which also allows us to install the firmware. The tool already comes with the firmware needed so it's just a matter of flashing it on to the dongle and installing Nordic's extension for Wireshark. After doing this, the dongle should appear as a capturing device in Wireshark, which allows us to capture the traffic like we would for IP packets for example.

# ZigBee packet analysis 

## Getting Wireshark to display and decrypt ZigBee packets
Getting the packets to be read by the sniffer is simple enough, however, ZigBee networks operate using 128-bit symmetric keys, which means that the default output in Wireshark will be unreadable. For this, we also have to configure the ZigBee protocol in Wireshark to have 2 additional keys, in order for the program to try and decrypt the content. 

The first one is known as the Link Key and usually can be found under the name ZigbeeAlliance09, which is a pre-shared key used for devices to join networks. The second one is the Network Key, which is generated by Zigbee2MQTT. The first key is only used to establish an encrypted tunnel with the ZigBee hub, in order to exchange the actual network key (the second key) which is used for the rest of the communications in the network. So by providing Wireshark with both of these keys, we are able to fully read the communications in the protocol.

## The environment

The ZigBee environment in which we did this testing consists of 3 components: a ZigBee compatible light bulb, in this case a Phillips Hue bulb, that only emits warm white light and only allows for brightness control, a Raspberry Pi with Zigbee2MQTT server running and a SONOFF ZBDongle-P based on the TI CC2652P chip. What Zigbee2MQTT does is convert MQTT messages into ZigBee packets, and due to the size of the project, there's an incredible amount of devices supported out of the box.

Firstly, we remove the light bulb in order to make it attempt to join the network. When it turns on into the join mode, the bulb will broadcast a beacon request, to which the hub will reply with a beacon containing its own PAN ID (PID), or EPID which is an extended version of a PID (16-bit vs 64-bit). In this setup, the hub will reply with an EPID and information regarding whether or not it allows the joining of new devices.

Now that the router and device have established a dialog, the router will try to send the device the network key, as all transmissions up until now have all been encrypted using the pre-shared key ZigbeeAlliance09. The router will send the key using a Transport Key message which contains the key that is exclusive to this network. After receiving the key, the device will now know that it is in the network and thus send a Device Announcement message, which contains its new short and extended addresses.

## Security concerns

The transport key being easily accessible using a simple Google search is a bit odd and unsecure since it makes sniffing a very easy task. An attacker will wait until a device initiates the joining process, at which point the attacker can easily decrypt the network key during the Transport Key message. One mitigation that the protocol employs is that the joining process is only available for a certain amount of minutes (Zigbee2MQTT defaults to 4 minutes. This makes the joining of new devices a more deliberate choice has the router has to have some sort of rule that decides when to allow joining, such as a person enabling it.

Replay attacks also jump into mind. Although the message may be encrypted, someone can still listen and try to resend a command for a device later on. To combat this, ZigBee devices keep a Frame counter that increment every transmission, and they also track the last-known counter of their neighbours. So if an attacker tries to send a message with a smaller frame counter, the ZigBee devices that would retransmit the packet will discard it instead. However, when the counter reaches its maximum value of 2^32 - 1, the device will no longer be able to send any more messages as the counter will reset back to 0. The only way to reset counters is to update to a new network key. This shouldn't be a problem for most networks as depending on the transmission rate of the device, the frame counter expiration can take over 100 years. source https://docs.digi.com//resources/documentation/digidocs/90002002/concepts/c_zb_frame_counter.htm?TocPath=Zigbee%20security%7CZigbee%20security%20model%7CNetwork%20layer%20security%7C_____1
